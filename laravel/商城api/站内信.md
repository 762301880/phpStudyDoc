## 大厂标准站内通知架构（双表懒加载方案，抖音 / 小红书 / B 站 / 电商通用）

核心思想：**内容只存 1 份，用户已读状态按需生成，全员推送绝不批量插几万行**

### 为什么大厂放单表结构

一条全员公告 → 10 万用户 = 10 万条重复标题 / 内容 / 图片，存储爆炸、发推送时数据库瞬间 IO 打满。

大厂统一用 **读扩散（懒加载）**：

1. 公共消息只存 1 行；
2. 用户不点消息页，完全不产生任何用户状态记录；
3. 只有用户打开通知、点击详情，才插入 / 更新已读；
4. 一万用户发公告，MySQL 只新增 1 条，无批量写入压力。

### 标准两张核心表（可直接落地）

#### 表 1：消息主体表 `fa_notice`（公共内容，一条公告仅一行）

存标题、正文、图片、跳转、推送范围、有效期，所有用户共享这一条

```sql
CREATE TABLE `notice` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '公告ID',
  `sender_id` int NOT NULL COMMENT '发送人ID（0=系统）',
  `type` tinyint NOT NULL DEFAULT 1 COMMENT '1系统通知 2业务推送 3互动 4活动公告',
  `tag` varchar(64) NOT NULL DEFAULT '' COMMENT '业务标识：novel_update/recharge等',
  `title` varchar(128) NOT NULL COMMENT '标题',
  `content` text NOT NULL COMMENT '正文',
  `image` varchar(255) DEFAULT '' COMMENT '封面图',
  `jump_route` varchar(128) DEFAULT '' COMMENT '跳转路由',
  `extra_data` json DEFAULT NULL COMMENT '业务扩展参数',
  `scope` tinyint NOT NULL DEFAULT 1 COMMENT '1=全员推送 2=指定用户 3=会员定向',
  `valid_start` datetime DEFAULT NULL COMMENT '生效时间',
  `valid_end` datetime DEFAULT NULL COMMENT '过期自动隐藏',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '发布时间',
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_tag_scope_time` (`tag`,`scope`,`valid_end`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='公共公告内容表';
```

#### 表 2：用户阅读状态表 `notice_read`（只存用户 + 公告 + 已读标记，体积极小）

```sql
CREATE TABLE `notice_read` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT,
  `notice_id` bigint unsigned NOT NULL COMMENT '关联fa_notice.id',
  `user_id` int NOT NULL COMMENT '当前用户ID',
  `is_read` tinyint NOT NULL DEFAULT 0 COMMENT '0未读 1已读',
  `read_at` datetime DEFAULT NULL COMMENT '阅读时间',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `is_delete` tinyint(1) NOT NULL DEFAULT '0' COMMENT '0 正常展示（区分未读 / 已读）  1 用户已删除，前端列表过滤掉',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_notice_user` (`notice_id`,`user_id`), -- 一条公告对一个用户只存一条
  KEY `idx_user_read` (`user_id`,`is_read`) -- 查询个人消息列表核心索引
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户公告阅读记录表';
```

#### 对应代码

```php
    public function getNoticeList(Request $request)
    {
        if (empty($this->user_id)) $this->success('站内信返回成功');

        $userId = $this->user_id;
        $limit = $request->param('limit', 10);

        $mr_is_delete_1 = MessageRead::IS_DELETE_1; // 1=已删除

        $list = Message::alias('m')
            ->field('m.*, IF(mr.is_read IS NULL, 0, mr.is_read) as is_read, mr.read_at, mr.is_delete')
            ->view(
                'user_message_read mr',
                'user_id,notice_id,is_read,read_at,is_delete',
                "mr.notice_id = m.id AND mr.user_id = {$userId}",
                'LEFT'
            )
            ->where(function ($query) use ($userId, $mr_is_delete_1) {
                // 全员消息 OR 指定用户消息
                $query->where('m.scope', Message::SCOPE_1)
                    ->whereOr(function ($q) use ($userId) {
                        $q->where('m.scope', Message::SCOPE_2)
                            ->where('mr.user_id', $userId);
                    });
            })
            // ✅【关键】统一过滤：当前用户已删除的消息，scope=1和scope=2共用
            ->where(function ($q) use ($mr_is_delete_1) {
                $q->where('mr.is_delete', '<>', $mr_is_delete_1)
                    ->whereOr('mr.is_delete', '0');
            })
            ->order('m.create_time desc')
            ->paginate($limit);
        $this->success('站内信返回成功', $list);
    }
```



### 完整业务流程（全员推送核心逻辑，重点）

#### 场景 1：发全站活动公告（scope=1 全员）

1. **写入notice 仅 1 条数据**，标记 `scope=1`，设置有效期；
2. **不操作 notice_read，不插入任何用户记录**（关键懒加载）；
3. 推送网关（WebSocket/APP 推送）单独给全量用户发推送通知，和数据库解耦；

用户打开消息列表时怎么查？

SQL 逻辑（左联判断已读）：

```sql
SELECT
  n.*,
  IF(r.is_read IS NULL OR r.is_read=0,0,1) AS is_read,
  r.read_at
FROM notice n
LEFT JOIN notice_read r 
  ON n.id = r.notice_id AND r.user_id = 登录UID
WHERE 
  n.scope = 1 
  AND NOW() BETWEEN n.valid_start AND n.valid_end
ORDER BY n.create_time DESC;
```

左联无记录 = 用户从未打开这条公告 → 默认展示**未读红点**；

左联有记录且 is_read=1 → 已读；

**用户点击公告标记已读**

执行幂等写入（防止重复记录）：

```sql
INSERT INTO notice_read (notice_id,user_id,is_read,read_at)
VALUES (公告ID,UID,1,NOW())
ON DUPLICATE KEY UPDATE is_read=1,read_at=NOW();
```

这条记录**只有当前点击的用户才会生成**，其余几万用户无数据，极大省存储。

####  场景 2：定向推送 100 个指定用户（scope=2）

1. notice 插入 1 条公告；
2. 异步队列批量 INSERT 100 条到 `notice_read`，默认 `is_read=0`；
3. 用户打开列表直接关联读取状态，无需左联判断。

### 大厂配套高性能优化（千万用户必备）

#### 1. Redis 缓存层（解决未读红点高频查询）

1. 每个用户维护 `user:unread:uid` 集合，存所有未读公告 ID；
2. 用户标记已读时同步删除集合内 ID；
3. 前端首页红点直接取集合长度，不用查 MySQL；
4. 全员公告发布时，不写全量用户缓存，**用户打开页面再懒加载更新缓存**。

#### 2. 分表 / 冷热数据分离（百万级用户标配）

- `fa_user_notice_read` 按 `user_id % 128` 水平分 128 张子表，单表行数控制在千万内；
- 超过 90 天过期公告：归档到历史库，线上只查近 3 个月有效公告；
- `fa_notice` 按月分区，过期公告分区直接冷备份。

#### 3. 推送解耦（MQ 异步，避免发公告拖垮接口）

发布公告不直接操作数据库批量插入，丢 RabbitMQ/Kafka 队列：

- 全员消息队列只写入 fa_notice，无批量 DB 写入；
- 定向消息队列异步批量 insert 用户阅读记录，限流保护数据库。

#### 4. 统计已读人数（业务需求）

- 粗略统计：用 Redis HyperLogLog 估算总已读用户，不查 DB；
- 精确导出：后台定时任务查询 `fa_user_notice_read` 对应 notice_id 的行数。

### 对比两种方案优劣，一眼看懂大厂为什么选双表

|     维度     |      你现在单表方案（写扩散）       |       大厂双表懒加载（读扩散）       |
| :----------: | :---------------------------------: | :----------------------------------: |
| 全员推送入库 | 1 万用户 = 插入 1 万行，内容全重复  |     仅插入 1 行公告，无用户记录      |
|   存储空间   |     海量冗余文本，半年数据爆炸      |       仅存 ID + 状态，占用极小       |
|  数据库压力  | 发全站推送瞬间大量 Insert，容易锁表 |       发布无批量写入，压力极低       |
|   已读逻辑   |   每条用户自带 is_read，查询简单    |    左联判断，配合 Redis 缓存优化     |
|  冷用户成本  |   哪怕从不打开消息，也占一条数据    |      从不打开 = 无任何存储开销       |
|   适用规模   |      用户几千、极少发全站公告       | 万 / 十万 / 百万级用户，高频运营推送 |

<script>
	import {
		CODE
	} from '@/constants/codeEnum.js'
	import api from "@/api"
	export default {
		data() {
			return {
				getSwipeOptions(item) {
					return [
						// {
						//   text: '取消',

				},
				activeTab: 1,
				tabList: [{
						name: '全部消息'
					},
					{
						name: '系统通知'
					}
				],
				messageList: []
			}
		},
		onLoad() {
			this.getMyNoticeList()
		},
		onShow() {
			this.getMyNoticeList()
		},
		methods: {
			onClick(e) {
				// console.log('点击了' + (e.position === 'left' ? '左侧' : '右侧') + e.content.text + '按钮')
				// 判断是右侧删除按钮
				if (e.position === 'right' && e.content.text === '删除') {
					const item = e.content.data // 把当前消息对象挂载到按钮data里传过来
					uni.showModal({
						title: '提示',
						content: '确定删除这条站内信？删除后无法恢复',
						success: async res => {
							if (res.confirm) {
								// 调用删除接口
								const delRes = await api.user.deleteOneMessage({
									id: item.id
								})
								if (delRes.code === CODE.SUCCESS) {
									// 本地过滤移除当前条目
									this.messageList = this.messageList.filter(v => v.id !== item.id)
									uni.showToast({
										title: '删除成功'
									})
								}
							}
						}
					})
				}
			},
			backPage() {
				uni.navigateBack()
			},
			async readAll() {
				const res = await api.user.readAllNotice({
					tab: this.activeTab
				})
				if (res.code === CODE.SUCCESS) {
					this.messageList.forEach(item => item.isRead = 1)
				}
			},
			tabChange() {
				this.getMyNoticeList()
			},
			async openDetail(item) {
				await api.user.readOneNotice({
					id: item.id
				})
				item.isRead = 1
				if (item.tag === 'U_S_LOG_V') { //用户搜索短剧日志
					const resObj = JSON.parse(item.extra_data)
					this.$shuaban.toinfo(resObj, 3)
				}
			},
			async getMyNoticeList() {
				this.messageList = []
				const res = await api.user.getNoticeList({
					tab: this.activeTab
				})
				const {
					code,
					data
				} = res
				if (code === CODE.SUCCESS && data && Array.isArray(data.data)) {
					// 修复：接口列表是 data.data 不是 data.data.data
					this.messageList = data.data
				}
			},
			formatTime(timeStr) {
				if (!timeStr) return ''
				const createTimestamp = new Date(timeStr.replace(/-/g, '/')).getTime()
				const now = Date.now()
				const diffSecond = Math.floor((now - createTimestamp) / 1000)

				if (diffSecond < 60 * 60) {
					return `${Math.floor(diffSecond / 60)}分钟前`
				} else if (diffSecond < 60 * 60 * 24) {
					return `${Math.floor(diffSecond / 3600)}小时前`
				} else if (diffSecond < 60 * 60 * 24 * 2) {
					return '昨天'
				} else {
					return `${Math.floor(diffSecond / (3600 * 24))}天前`
				}
			}
		}
	}
</script>
