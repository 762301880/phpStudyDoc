## 	分销模式

> 核心业务模型（先定规则，不然后面全乱）

###  分销模式（必须先选）

常见三种：

#### ✅ 一级分销（最简单，推荐）

- A 推荐 B
- B 下单 → A 拿佣金

 👉 优点：简单、合规、容易算账
 👉 缺点：裂变弱

------

#### ⚠️ 二级分销（常见）

- A → B → C
- C 下单 → B + A 都有佣金

👉 问题：

- 关系链复杂
- 佣金计算容易错
- 风控难（刷层级）

------

#### ❌ 多级分销（不建议）

- 容易踩法律红线（类似传销）

✔️ 建议：**最多二级，最好一级起步**

##  佣金计算逻辑

### 下单后触发：

```shell
用户C下单
  ↓
找到C的 inviter（B）
  ↓
B拿一级佣金
  ↓
如果有二级 → 找B的 inviter（A）
  ↓
A拿二级佣金
```

### 示例计算：

商品：100元
 一级佣金：10%
 二级佣金：5%

```shell
C买 → 100元

B（一级） → 10元  
A（二级） → 5元
```

正确计算方式（行业通用）

订单金额：100 元

- 一级（B）：
   `100 × 10% = 10 元`
- 二级（A）：
   `100 × 5% = 5 元`

### 总佣金是多少？

> 分销系统默认是：
>  **“每一层都基于订单金额 independently 计算佣金”**

```
总发放佣金 = 10 + 5 = 15 元
```

也就是：

> ✅ **相当于平台拿出 100 的 15% 用来分佣**

但注意一点：

👉 **不是“先给 B 10%，再从 B 里抽 5% 给 A”**

- 佣金不要“实时到账”
- 必须有状态流转：

👉 防止退款导致财务错误

## 关键技术点

### 1.推荐关系绑定（坑点）

用户注册时：

```
?invite_code=xxx
```

👉 必须只绑定一次，不可修改

### 2. 防刷机制

否则分销=送钱

要做：

- 同 IP 限制
- 同设备限制
- 自买自返检测
- 异常订单识别

### 3. 佣金异步计算（重要）

不要在下单接口直接算佣金！

👉 用消息队列（你之前用 RabbitMQ，很适合）

流程：

```
下单成功
  ↓
发送 MQ 消息
  ↓
佣金服务消费
  ↓
写入 commission 表
```

👉 好处：

- 解耦
- 防止接口变慢
- 可重试

### 4. 分销关系查询优化

不要递归查！

用 `path`：

```sql
SELECT * FROM user WHERE id IN (path拆分)
```

### 5. 提现系统（容易出事故）

需要：

- 最低提现金额
- 手续费
- 审核机制
- 打款记录

##  补充

### 分销如何分等级

**如何在 A → B → C 这种推荐关系里，明确谁是一级、谁是二级，并在代码里稳定识别出来**。

**关系定义**

```php
A → B → C
```

含义：

- A 是 B 的邀请人
- B 是 C 的邀请人

那么：

| 角色 | 对 C 来说的层级 |
| ---- | --------------- |
| B    | 一级分销        |
| A    | 二级分销        |

👉 记住一句话：

> **“离下单用户最近的是一级，再往上是二级”**

#### 核心字段设计（决定你怎么判断）

用户表关键字段：

```sql
id           -- 当前用户
inviter_id   -- 直接上级(父节点  如果自己就是父节点null)
path         -- 推荐链，例如：A/B/C
```

**用 A / B / C 举例说明**

假设有这样一条推荐关系：

```
A → B → C
```

意思是：

- A 推荐了 B
- B 推荐了 C

数据会这样存：

用户 A（最顶级，没有上级）

```php
id = A
inviter_id = NULL
path = A
```

用户 B（A 推荐）

```php
id = B
inviter_id = A
path = A/B
```

用户 C（B 推荐）

```php
id = C
inviter_id = B
path = A/B/C
```

**path 是怎么生成的?**

规则很简单：

```php
当前用户    path = 上级用户 path + "/" + 当前用户 id
```

比如：

- B 的 path = A + "/" + B → `A/B`
- C 的 path = A/B + "/" + C → `A/B/C`





#### 代码判断逻辑（重点）

假设：
 当前下单用户 = `C`

✅ 方法1：只用 inviter_id（最简单)

```php
// C
level1 := C.inviter_id   // B

// B
level2 := B.inviter_id   // A
```

👉 判断：

```php
B = 一级
A = 二级
```

✅ 方法2：用 path（推荐，用于复杂场景）

假设：

```
A.id = 1
B.id = 2
C.id = 3

C.path = "1/2"
```

解析：

```
path 最后一个 = 直接上级 = 一级（B）
path 倒数第二个 = 二级（A）
```

#### 佣金分配逻辑（结合层级）

```php
C 下单
 ↓
一级（B）拿佣金
 ↓
二级（A）拿佣金（如果存在）
```

**完整流程**

```php
1. 用户 C 下单
2. 查 C.inviter_id → B
   → 给 B 一级佣金

3. 查 B.inviter_id → A
   → 给 A 二级佣金
```

## 关键限制（必须做）

### ❗ 1. 限制层级深度

```
最多两级，不要无限往上找
```

------

### ❗ 2. inviter_id 必须固定

```
用户一旦绑定上级 → 永远不能改
```

------

### ❗ 3. 防止自己给自己返佣

```
if (A == C) → 不发佣金
```
