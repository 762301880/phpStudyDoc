# 分销系统核心字段设计（通用版）

我给你整理**企业级、可直接落地、兼容电商 / 零售 / SAAS**的分销系统字段设计，包含**用户、分销关系、订单、佣金、提现、商品**6 大核心模块，无冗余、可直接建表使用。

## 一、整体设计原则

1. **层级清晰**：支持 1~3 级分销（合规主流）
2. **关系可追溯**：上级、团队、来源链路完整
3. **佣金可结算**：订单、佣金、提现闭环
4. **防篡改**：关键金额、状态不可随意修改
5. **易扩展**：支持等级、权限、活动

# 二、核心表结构 + 字段设计（MySQL 风格）

## 1. 用户表（user）

存储所有普通用户 + 分销商

```sql
CREATE TABLE `user` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '用户ID',
  `nickname` varchar(50) DEFAULT '' COMMENT '昵称',
  `mobile` varchar(20) DEFAULT '' COMMENT '手机号',
  `avatar` varchar(255) DEFAULT '' COMMENT '头像',
  `is_distributor` tinyint DEFAULT 0 COMMENT '是否分销商 0否 1是',
  `distributor_level` tinyint DEFAULT 0 COMMENT '分销等级：0普通用户 1初级分销商 2高级分销商',
  `balance` decimal(12,2) DEFAULT 0.00 COMMENT '可提现余额',
  `frozen_amount` decimal(12,2) DEFAULT 0.00 COMMENT '冻结金额',
  `total_commission` decimal(12,2) DEFAULT 0.00 COMMENT '累计佣金',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_mobile` (`mobile`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

###  字段解析

#### distributor_level 是什么情况下升级的

`distributor_level` = **用户的分销商身份等级**

不是注册就固定死，**可以后期升级、降级**，不是一创建就定终身。

- `0`：**普通用户**，不是分销商，没有分销权限、拿不到佣金
- `1`：**一级分销商**，有推广资格，能拿一级佣金
- `2`：**高级分销商**/ 代理，佣金比例更高、可能能拿二级 / 三级团队佣金

####  疑问：难道创建账号就知道自己等级了？

**不是！绝对不是注册就定死等级**

真实业务流程是这样：

1. **刚注册用户**

   默认 `distributor_level = 0`

   就是普通路人账号，不能推广、没有分销资格。

2. **满足条件后才升级**

常见升级条件：

- 自己付费开通分销商
- 消费满多少金额自动升级
- 拉够多少下级自动升级
- 后台管理员手动给设置等级

​    满足条件后，**后台代码直接更新这个字段**：

​     从 0 → 1 ，或者 1 → 2

3. 也可以**降级**

不达标、违规，管理员或系统自动把等级改回 0/1

#### `is_distributor`：**开关**，能不能做分销

> 不管 level 是几，都按普通用户算

#### frozen_amount 冻结金额怎么来、什么时候产生、什么时候解冻

一、先定义

- `balance`：**可提现余额**（随时能申请提现）
- `frozen_amount`：**冻结金额**（看得见、不能提、不能花，被锁住了）

冻结金额 3 个核心来源（分销系统最常见）

来源 1：订单佣金未到结算期 → 先冻结

分销行业通用规则：

用户下单后，**不能马上给你结算佣金**，要等：

- 确认收货
- 售后退货期过完（一般 7 天 / 15 天无退款）

流程：

1. 下级下单产生佣金
2. 系统**不进可提现余额 balance**
3. 直接加到 **frozen_amount 冻结金额**
4. 等到结算周期到、无退货退款
5. 从冻结金额扣掉，转到 **balance 可提现余额**

👉 这是**冻结金额最主要的来源**

来源 2：用户申请提现 → 金额立刻冻结

用户点「申请提现」：

1. 从 `balance` 可提现余额 扣钱
2. 加到 `frozen_amount` 冻结
3. 进入后台待审核
   - 审核通过打款：冻结金额扣除，清零
   - 审核驳回：冻结金额退回 balance

来源 3：违规 / 封号 / 限制分销 → 后台手动冻结

管理员后台操作：

- 违规推广
- 刷单作弊
- 账号风控

直接把部分 / 全部余额**拉入冻结**，禁止提现。

三、资金流转全过程（超通俗）

1. 下级下单 → 生成佣金
2. 佣金先进 **冻结金额 frozen_amount**
3. 过了售后结算期 → 自动解冻
4. 解冻后转入 **可提现余额 balance**
5. 申请提现 → balance 扣钱 → 进入 frozen 冻结待打款
6. 打款完成 → frozen 减掉

四、举个数字例子

你现在：

- balance：100 元（可提）
- frozen_amount：0 元

1. 下级下单，产生佣金 **50 元**

   → frozen_amount 变成 **50**

   → balance 还是 100

   

2. 7 天售后过了，无退款

   → frozen 变回 0

   → balance 变成 **150**

   

3. 你申请提现 80 元

   → balance：70

   → frozen：80（锁住待审核）

   

五、为什么一定要有冻结金额？

如果没有冻结：

- 用户刚下单立马拿佣金提现
- 后面用户退货退款了
- 平台钱亏了、追不回来

`frozen_amount` 就是**防止退货刷单、提前提现**的风控字段。



## 2. 分销关系表（distributor_relation）

**分销系统最核心表**：记录谁推荐谁、上下级关系、团队链路

```sql
CREATE TABLE `distributor_relation` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `user_id` bigint NOT NULL COMMENT '当前用户ID',
  `parent_id` bigint NOT NULL COMMENT '直接上级ID',
  `level_1_id` bigint DEFAULT 0 COMMENT '1级上级',
  `level_2_id` bigint DEFAULT 0 COMMENT '2级上级',
  `level_3_id` bigint DEFAULT 0 COMMENT '3级上级',
  `team_path` varchar(255) DEFAULT '' COMMENT '团队链路 0-1001-1002-1003',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_user_id` (`user_id`),
  KEY `idx_parent_id` (`parent_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='分销上下级关系表';
```

## 3. 分销订单表（distributor_order）

订单产生后，自动生成分销佣金记录

```sql
CREATE TABLE `distributor_order` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `order_no` varchar(64) NOT NULL COMMENT '订单号',
  `user_id` bigint NOT NULL COMMENT '下单用户ID',
  `order_amount` decimal(12,2) NOT NULL COMMENT '订单实付金额',
  `commission_rate` decimal(6,2) DEFAULT 0.00 COMMENT '佣金比例',
  `total_commission` decimal(12,2) DEFAULT 0.00 COMMENT '订单总佣金',
  `status` tinyint DEFAULT 0 COMMENT '0待结算 1已结算 2已取消',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `settle_time` datetime DEFAULT NULL COMMENT '结算时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_order_no` (`order_no`),
  KEY `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT=utf8mb4 COMMENT='分销订单表';
```

## 4. 佣金明细表（commission_record）

CREATE TABLE `commission_record` (

# 三、核心业务逻辑（极简说明）

1. **用户注册 → 写入分销关系表**（绑定上级）
2. **用户下单 → 生成分销订单 → 计算佣金**
3. **佣金按 1/2/3 级分别入账**
4. **订单完成 → 佣金解冻 → 可提现**
5. **分销商申请提现 → 冻结余额 → 审核打款**
