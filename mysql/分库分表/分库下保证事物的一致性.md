## MQ实现分库下的事务一致性

一、核心思想（一句话）

**本地事务 + 消息最终一致性**

> 先把“自己的库”保证成功，再用 MQ 通知别的库去完成

二、完整流程

```shell
订单服务（订单库）
   ↓ 本地事务
订单 + 消息表(message_log)
   ↓
定时任务/投递器
   ↓
RabbitMQ
   ↓
用户服务消费
   ↓
扣余额
   ↓
失败重试 / 死信队列
```

## Laravel 落地实现

 第一步：消息表（核心）

```sql
message_logs
------------
id
topic
body
status (pending/sent/failed)   -- pending:待处理  sent:投递  failed:失败
retry_count  --重试次数
created_at   -- 创建时间
```

### 字段解释

`topic`

消息主题 / routing key / 队列分类

比如你之前做 RabbitMQ：

- dota
- csgo
- lol

可能存：

```
topic = 'lol'
```

表示这条消息是发往哪个业务模块。

### `body`

消息内容

通常存 JSON：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "amount": 500
}
```

表示真正要发送的数据。

比如：

- 下单消息
- 支付消息
- 发货消息

### `status`(消息当前状态)

你表里定义了：

#### `pending`(待处理)

消息刚创建，还没真正发送到 MQ：

> 数据库有记录
> 但还没投递

#### `sent`(已投递成功)

表示：

- 已经成功发送到 RabbitMQ/Kafka
- broker 接收成功

注意：

这 **不一定代表消费者处理成功**

只是“投递成功”。

#### `failed`(发送失败)

例如：

- RabbitMQ 宕机
- 网络异常
- producer 报错

消息没成功发出去。

### `retry_count`(重试次数)

比如第一次发送失败：

```
retry_count = 1
```

第二次失败：

```
retry_count = 2
```

用于：

- 限制最大重试次数
- 防止无限死循环

例如：

```php
if ($retry_count > 3) {
    status = 'failed';
}
```



### 整体流程

比如用户下单：

1. 用户下单成功
2. 插入 `message_logs`

```
status = pending
```

1. 定时任务扫描 pending
2. 投递 RabbitMQ
3. 成功：

```
status = sent
```

失败：

```
status = failed
retry_count + 1
```

### 下单（本地事务 + 写消息）

```php
DB::transaction(function () use ($orderData) {

    // 1. 创建订单
    $orderId = DB::table('orders')->insertGetId([
        'user_id' => $orderData['user_id'],
        'amount' => $orderData['amount'],
        'status' => 'pending'
    ]);

    // 2. 写消息表（关键！）
    DB::table('message_logs')->insert([
        'topic' => 'order_created',
        'body' => json_encode([
            'order_id' => $orderId,
            'user_id' => $orderData['user_id'],
            'amount' => $orderData['amount']
        ]),
        'status' => 'pending',
        'retry_count' => 0
    ]);
});
```

到这里：
订单一定存在
消息一定存在（不会丢！）

**投递消息（异步任务）**

```php
class SendMessageJob implements ShouldQueue
{
    public function handle()
    {
        $messages = DB::table('message_logs')
            ->where('status', 'pending')
            ->limit(100)
            ->get();

        foreach ($messages as $msg) {
            try {
                app('mq')->publish($msg->topic, $msg->body);

                DB::table('message_logs')
                    ->where('id', $msg->id)
                    ->update(['status' => 'sent']);

            } catch (\Exception $e) {

                DB::table('message_logs')
                    ->where('id', $msg->id)
                    ->increment('retry_count');
            }
        }
    }
}
```

👉 可以用：

- Laravel Queue
- 或 cron 定时跑

**RabbitMQ 消费者（用户服务**

```php
consume('order_created', function ($msg) {

    $data = json_decode($msg, true);

    // 幂等判断（必须！）
    if (DB::table('consume_logs')->where('msg_id', $data['order_id'])->exists()) {
        return;
    }

    DB::transaction(function () use ($data) {

        // 扣余额
        DB::table('user_balances')
            ->where('user_id', $data['user_id'])
            ->decrement('balance', $data['amount']);

        // 记录消费
        DB::table('consume_logs')->insert([
            'msg_id' => $data['order_id']
        ]);
    });
});
```

必须解决的3个问题

1. 消息丢失

解决：
 👉 **消息入库（message_logs）**

2. 消息重复消费

解决：
 👉 **幂等表（consume_logs）**

```
if (exists) return;
```

3. 消费失败

解决：

 重试机制

```
retry_count < 5
```

死信队列（DLQ）

失败太多 → 丢到死信队列人工处理

##   如果有三个库呢   订单库   扣 库存   然后 发货  三个库

**从“两个库一致性”升级到“三个库（订单 + 库存 + 发货）一致性”**

  结论：

> **MQ方案天然支持多个库，本质就是“事件链” or “事件驱动”**
> 不是一次搞定，而是**一步一步传递，最终收敛一致**

分库 + MQ 场景里，正确模型是：

```php
订单 → 库存 → 发货（链式驱动）
```

Step 1️⃣：订单服务（起点）

```php
DB::transaction(function () {

    $orderId = createOrder();

    saveMessage('order_created', [
        'order_id' => $orderId
    ]);
});
```

Step 2️⃣：库存服务（消费 order_created）

```php
consume('order_created', function ($msg) {

    if (isConsumed($msg['order_id'], 'stock')) return;

    DB::transaction(function () use ($msg) {

        deductStock($msg['order_id']);

        markConsumed($msg['order_id'], 'stock');

        // 👉 关键：继续往下发事件
        saveMessage('stock_deducted', [
            'order_id' => $msg['order_id']
        ]);
    });
});
```

Step 3️⃣：发货服务（消费 stock_deducted）

```php
consume('stock_deducted', function ($msg) {

    if (isConsumed($msg['order_id'], 'delivery')) return;

    DB::transaction(function () use ($msg) {

        createShipment($msg['order_id']);

        markConsumed($msg['order_id'], 'delivery');

        saveMessage('order_completed', [
            'order_id' => $msg['order_id']
        ]);
    });
});
```

关键点你一定要抓住（核心理解🔥）

✅ 1. 不是“一个事务”，是“事务链”

每个服务只保证：

```
自己的数据库事务 ✔
+ 发出下一步消息 ✔
```

✅ 2. 每一步都必须“可重试 + 幂等”

```php
if (alreadyProcessed(order_id, step)) return;
```

否则：

- MQ重复投递 ❌
- 网络抖动 ❌
   → 数据直接炸

✅ 3. 每一步都要“本地消息表”

每个库都有：

```php
message_logs
consume_logs # 消耗
```

 **不是一张全局表！是每个服务一套！**

四、那如果中间失败怎么办？（重点⚠️）

比如：

```
订单 ✔
库存 ❌
发货 ❌
```

怎么办？

方案一：自动重试（最常用）

库存失败：

- MQ 自动 retry
- 或定时任务扫 message_log

 方案二：补偿（Saga 思想）

如果库存一直失败：

```php
cancelOrder($orderId);
```

订单回滚（状态改为 canceled）

方案三：人工兜底（生产必备）

- 死信队列（DLQ）
- 后台手动处理

### 进阶：状态机设计（非常关键🔥）

订单表不要简单 status，要这样：

```
orders
------
id
status:
    pending
    stock_deducted
    shipping
    completed
    canceled
```

每一步更新状态：

```
订单创建 → pending
库存成功 → stock_deducted
发货成功 → completed
```

👉 这样你可以：

- 查异常订单
- 做补偿
- 做重试
