# 常见的分表方式

## 一、按拆分维度分类

## 1️⃣ 水平分表（最常用）

👉 **按行拆分**（数据分散到多个表）

### ✔️ 例子：按 user_id 取模

```
user_0
user_1
user_2
user_3
$table = 'user_' . ($user_id % 4);
```

### ✔️ 特点

- 每张表结构一样
- 数据量被均摊
- 解决**单表数据量过大问题（千万/亿级）**

### ✔️ 适用场景

- 用户表、订单表、日志表

## 2️⃣ 垂直分表

👉 **按列拆分**（字段拆开）

### ✔️ 例子：用户表拆分

```shell
user_base（核心字段）
id, name, age

user_ext（扩展字段）
id, avatar, bio, address
```

### ✔️ 特点

- 减少行宽
- 提高缓存命中率
- 热字段 & 冷字段分离

### ✔️ 适用场景

- 字段很多（几十上百列）
- 大字段（text/blob）

##  🧠 一句话先理解

👉 **垂直分表 = 把“一行太胖的数据”拆成两张表 ** 拆列（字段）

真实案例：用户表太“胖”了

原始表:

```sql
user (
  id BIGINT,
  name VARCHAR(50),
  age INT,
  phone VARCHAR(20),
  avatar TEXT,
  bio TEXT,
  address TEXT,
  create_time DATETIME
)
```

👉 问题来了：

- avatar / bio / address 是 **大字段（text）**
- 但大多数查询只用：
  👉 id + name + age

⚠️ 结果：

- 每次查用户列表，都把一堆 **没用的大字段一起读出来**
- IO暴涨，缓存命中低

✂️ 垂直拆分（核心）

拆成两张表：

✅ 1️⃣ 核心表（高频访问）

```sql
user_base (
  id BIGINT,
  name VARCHAR(50),
  age INT,
  phone VARCHAR(20),
  create_time DATETIME
)
```

✅ 2️⃣ 扩展表（低频访问）

```sql
user_ext (
  id BIGINT,
  avatar TEXT,
  bio TEXT,
  address TEXT
)
```

🔥 查询对比（重点）

❌ 拆分前（浪费IO）

```php
SELECT * FROM user WHERE id = 1;
```

👉 实际发生：

- 把 avatar、bio 这些大字段也读出来了（浪费）

✅ 拆分后（高效）

### 场景1：用户列表（高频）

```
SELECT id, name, age FROM user_base LIMIT 10;
```

👉 优点：

- 表更小（行更窄）
- 一页能装更多数据（MySQL页16KB）
- 查询更快 🚀

场景2：查看用户详情（低频）

```php
SELECT b.*, e.avatar, e.bio
FROM user_base b
LEFT JOIN user_ext e ON b.id = e.id
WHERE b.id = 1;
```

👉 只在需要时才查大字段

## 📊 本质提升在哪里？

### 1️⃣ 减少行宽（核心点）

👉 原来一行可能 2KB，现在变 200B

➡️ 一页能放更多行 → IO减少

------

### 2️⃣ 缓存命中率更高

👉 Redis / Buffer Pool

- 原来：缓存一条=很大
- 现在：缓存更多“有效数据”

------

### 3️⃣ 热冷分离（非常关键🔥）

| 类型   | 字段        |
| ------ | ----------- |
| 热数据 | id, name    |
| 冷数据 | avatar, bio |

👉 热数据留在小表里 → 查询飞快

## 二、按分表策略（路由方式）分类（重点面试）

## 3️⃣ 取模分表（Hash 分表）

👉 最经典

```
$table = user_id % 8;
```

### ✔️ 优点

- 数据分布均匀

### ❌ 缺点

- 扩容困难（8 → 16 要迁移数据）

## 4️⃣ 范围分表（Range）

👉 按区间拆

### ✔️ 例子：按时间

```
order_202401
order_202402
order_202403
```

或

```
order_0 (0~1w)
order_1 (1w~2w)
```

### ✔️ 优点

- 方便范围查询（between）

### ❌ 缺点

- 热点问题（最近数据全在一个表）

## 5️⃣ 一致性 Hash 分表

👉 解决扩容问题

### ✔️ 思路

- hash → 环
- 节点少迁移

### ✔️ 优点

- 扩容平滑

### ❌ 缺点

- 实现复杂

------

## 6️⃣ 地理 / 业务维度分表

👉 按业务属性拆

### ✔️ 例子

```
order_beijing
order_shanghai
order_tokyo
```

或者

```
order_food
order_hotel
order_ticket
```

### ✔️ 优点

- 业务隔离
- 降低耦合

### ❌ 缺点

- 跨维度查询麻烦

------

## 7️⃣ 按冷热数据分表

👉 热数据 & 冷数据分离

### ✔️ 例子

```
order_hot（最近3个月）
order_history（历史数据）
```

### ✔️ 优点

- 提高查询性能
- 减少索引压力
