### 方法二

> 在MySQL中，如果你需要在按日期分组统计订单时补充缺失的日期，可以采用以下步骤来实现：
>
> 1. **创建一个日期序列表**：这是一个包含所有可能日期的表，用于确保即使没有订单的日期也能显示。
> 2. **LEFT JOIN日期序列表和订单表**：通过左连接将日期序列表和订单表连接起来，以确保所有日期都包含在最终结果中。
> 3. **使用聚合函数**：例如`COUNT`或`SUM`，来计算每个日期的订单数。
>
> 假设你的订单表名为`orders`，结构如下：

```sql
orders (
    id INT,
    order_date DATE,
    amount DECIMAL(10, 2)
)
```



### 1. 创建日期序列表

>  首先，创建一个包含所有日期的辅助表。假设你需要统计2023年1月1日至2023年12月31日的订单：
>
> `//`是MySQL中用于定义和结束存储过程的分隔符，因为默认的分隔符`;`在存储过程中会造成歧义，所以临时使用`//`作为新的分隔符。这里是带注释的存储过程代码：

```sql
-- 创建日期序列表
CREATE TABLE dates (
    date DATE PRIMARY KEY
);

-- 更改分隔符，避免存储过程中的语句与默认的分隔符;冲突
DELIMITER //

-- 创建填充日期序列表的存储过程
CREATE PROCEDURE FillDates(IN startDate DATE, IN endDate DATE)
BEGIN
    DECLARE currentDate DATE; -- 声明一个变量，用于循环中的当前日期
    SET currentDate = startDate; -- 初始化currentDate为开始日期
    -- 使用WHILE循环从开始日期填充到结束日期
    WHILE currentDate <= endDate DO
        INSERT INTO dates (date) VALUES (currentDate); -- 插入当前日期到dates表
        SET currentDate = DATE_ADD(currentDate, INTERVAL 1 DAY); -- 将currentDate增加一天
    END WHILE;
END//

-- 恢复默认的分隔符
DELIMITER ;

-- 调用存储过程，填充日期表从2023-01-01到2023-12-31
CALL FillDates('2023-01-01', '2023-12-31');

```

### 2. LEFT JOIN 和 GROUP BY

将日期表和订单表进行左连接，并按日期分组进行统计：

```sql
SELECT
    d.date,
    COUNT(o.id) AS order_count,
    IFNULL(SUM(o.amount), 0) AS total_amount
FROM
    dates d
LEFT JOIN
    orders o ON d.date = o.order_date
GROUP BY
    d.date
ORDER BY
    d.date;
```

## 查询为零(0)的数据记录

> 开发过程中因为有可能接手别人写的代码，一般数据库中的某种状态有的人喜欢用**0**代替，我们都知道判断查询条件时候外层
>
> 需要有一个为空判断一般情况下我们喜欢用php的**empty()或者isset()**函数去判断,这两个函数有个坑就是为**空或者0**的条件也会返回**真**,
>
> 但是为空情况下我们不希望查询,为0的情况下再查询,这时候就很难受了，所以这里我采用php7的**??**条件判断空(这里为**0**也会返回**0**),为了更安全
>
> 第二步骤我也做了一层判断是否是在返回规定的状态数组之内绝对安全
>
> **最好的解决方案**
>
> 设计数据库状态的时候千万不要用**0**这个状态。例如禁用状态你可以用负数**-1**,

```php
 $status = $searchParams['status'] ?? "";
 $status = $status != null && in_array($status, PositionModel::getStatusIds()) ? $searchParams['status'] : "";
 # 查询条件可以这样写
 if ($status != null) {
       $query->where('status', $status);
```

```mysql
SELECT t1.*
```

## 如何结合MySQL查询实现用户及其跟进记录的数量统计

> A表与B表的查询及用户跟进数量统计
>
> 场景:  
> 我有两张表   A表 用户表 users  B表 用户跟进表 users_follow   现在我需要显示A表的所有字段且多包含一个字段用户跟进数量统计字段 user_follow_count      

```mysql
SELECT 
    A.*, 
    COUNT(B.id) AS user_follow_count
FROM 
    users A
LEFT JOIN 
    users_follow B ON A.id = B.user_id
GROUP BY 
    A.id;
```

**更多种mysql实现语法**

```mysql
-- 使用子查询：

SELECT a.*, (SELECT COUNT(*) FROM users_follow WHERE user_id = a.id) AS follow_count
FROM users a;

SELECT a.*, (SELECT COUNT(*) FROM users_follow WHERE user_id = a.id HAVING COUNT(*) >= 2) AS follow_count 
FROM users a; -- 加查询条件count>2



-- 使用LEFT JOIN：需要查询A表所有跟进记录不为空情况下
SELECT a.*, COUNT(b.id) AS follow_count
FROM users a
LEFT JOIN users_follow b ON a.id = b.user_id
GROUP BY a.id;


-- 使用INNER JOIN： 需要统计A表有用户跟进记录情况下
SELECT a.*, COUNT(b.id) AS follow_count
FROM users a
INNER JOIN users_follow b ON a.id = b.user_id
GROUP BY a.id;

-- 使用窗口函数(仅适用于支持窗口函数的数据库，如MySQL 8.0+、PostgreSQL等）：)

SELECT a.*, COUNT(b.id) OVER (PARTITION BY a.id) AS follow_count
FROM users a
LEFT JOIN users_follow b ON a.id = b.user_id;
```

**补充**

**需要排序 user_follow_count 字段怎么办**

```mysql
SELECT 
    users.*, 
    COUNT(users_follow.id) AS user_follow_count
FROM 
    users
LEFT JOIN 
    users_follow ON users.id = users_follow.user_id
GROUP BY 
    users.id
ORDER BY 
    user_follow_count DESC;

```

>
> 具体来说，我们首先执行内部查询，它会返回一个包含机构ID、机构名称和对应计数的结果集合。然后，外部查询对这个结果集进行处理，例如在本例中使用了SUM函数来计算内部查询中计数的总和，从而得到了一个汇总统计值。

> 这种写法相对灵活，可以根据需要对内部查询和外部查询进行调整，以满足具体的数据处理需求。

> 你可以使用 MySQL 的 UPDATE 语句来实现这个目标。你可以编写一条 SQL 查询来更新你的表，将符合条件的来源字段值更新为你所需要的值。下面是一个示例：
>
> ```sql
> -- CASE 表示函数开始，END 表示函数结束。如果 condition1 成立，则返回 result1, 如果 condition2 成立，则返回 result2，当全部不成立则返回 result，而当有一个成立之后，后面的就不执行了。
> 
> SELECT
> CASE
>     WHEN 1 > 0 THEN '1 > 0' 
>     WHEN 2 > 0 THEN '2 > 0'
>     ELSE '3 > 0'
> END as `判断业务`
> 
> -- 满足条件一中断执行 返回
> 1 > 0
> ```



> 这个语句是一个SQL的UPDATE语句，用于更新数据库表中的数据。下面是它的执行过程：
>
> 1. 首先，该语句指定了要更新的表名（your_table_name），你需要将"your_table_name"替换为你要操作的实际表名。
> 2. 然后，使用SET关键字来设置要更新的列和新的值。在这个例子中，我们要更新名为"source"的列。
> 3. 接下来，使用CASE语句来确定每个条件下的新值。根据不同的条件，将"source"列的值进行相应的更新。具体来说：
>    - 如果"source"的值为27，则将其更新为28；
>    - 如果"source"的值为28，则将其更新为29；
>    - 如果"source"的值为29，则将其更新为30；
>    - 如果"source"的值为30，则将其更新为27；
>    - 如果"source"的值不满足上述任何条件，则保持原值不变。
> 4. 最后，使用WHERE子句来限制只更新满足条件的行。在这个例子中，只有当"source"列的值为27、28、29或30时才会被更新。
>

## mysql order by排序为什么需要排序的字段可以是别名字段

> 在 MySQL 中，`ORDER BY` 子句用于对查询结果进行排序。你提到的别名字段指的是在查询中使用 `AS` 关键字为字段创建的别名。
>
> MySQL 允许在 `ORDER BY` 子句中使用别名字段进行排序。这是因为 SQL 的执行顺序中，`ORDER BY` 是在 `SELECT` 语句的最后阶段进行处理的。这意味着在 `SELECT` 中定义的别名已经存在，可以被 `ORDER BY` 访问和使用。
>
> 让我们通过一个例子来说明：
>
> 假设你有一个 `users` 表，其中包含 `id`、`name` 和 `age` 列。现在你想对查询结果按照 `age` 列的倒序进行排序，并且你希望在结果中显示 `age` 列的别名 `年龄`。你可以这样做：

```mysql
SELECT name, age AS 年龄
FROM users
ORDER BY 年龄 DESC;
```

> 在这个查询中，`age AS 年龄` 创建了一个名为 `年龄` 的别名字段。然后，`ORDER BY` 子句可以使用这个别名字段来对结果进行排序，因为在执行 `ORDER BY` 的时候，`年龄` 字段已经被计算出来了。
>
> 这种方法使得查询语句更具可读性和易用性。如果你尝试在 `ORDER BY` 中直接使用原始的 `age` 列名，也是可以的，但在包含较复杂计算或多表连接的查询中，使用别名会更清晰和易于理解。



**详细解释**

> 当你在一个 SQL 查询中定义了别名，比如使用 `AS` 关键字为某个字段创建了一个新的名称，这个别名实际上就是在查询结果集中为该字段创建了一个新的标签或标识符。这样，你就可以通过这个新的名字来引用原始字段，而不必每次都使用原始的字段名称。
>
> 当你使用 `ORDER BY` 子句对结果进行排序时，实际上是在告诉数据库根据某个字段的值来排列结果。在这个排序过程中，数据库引擎需要知道要排序的是哪个字段。这就是为什么在 `ORDER BY` 中你通常会指定原始的列名，因为数据库需要知道按照哪个字段进行排序。
>
> 但是，当你在查询中使用了别名时，这个别名是在查询的结果集中存在的，因此在 `ORDER BY` 中也可以使用这个别名。这是因为 `ORDER BY` 是在结果集生成之后进行的操作。也就是说，别名在结果集生成之后才会有，所以在 `ORDER BY` 中使用别名并不会导致错误。
>
> 让我们通过一个例子来说明这个概念。假设我们有一个 `students` 表，包含 `name` 和 `score` 两列，现在我们想查出每个学生的得分，并按照得分高低进行排序。我们可以这样做：
>
> 首先，我们查询学生的得分，并为得分列创建一个别名 `总分`：

```mysql
SELECT name, score AS 总分  FROM students;
```

这会返回一个结果集，其中包含每个学生的姓名和总分列。现在，如果我们想按照这个 `总分` 列对结果进行排序，我们可以继续这样做：

```mysql
SELECT name, score AS 总分
FROM students
ORDER BY 总分 DESC;
```

> 在这个示例中，`ORDER BY` 子句使用了 `总分` 这个别名字段来对结果进行降序排序。这是因为在 SQL 执行过程中，`ORDER BY` 是在 `SELECT` 查询结果生成之后进行处理的。
>
> 总的来说，使用别名可以使查询更易读、易懂，而在 `ORDER BY` 子句中使用这些别名则是合法的，因为数据库已经知道了这些别名在结果集中的存在。

## mysql in 查询 如果in的字段为空为什么不会查询出来

> 在MySQL中，当使用IN语句进行查询时，**空值（NULL）不会被匹配，即不会导致返回包含空值的记录**。
>
> 详细来说，当你在IN语句中列出一系列值进行匹配时，如果这些值中包含空值（NULL），那么MySQL实际上不会将这个空值考虑在内。这意味着，如果你期望通过IN语句来筛选出那些字段值为NULL的记录，这样的查询是**不会如你所愿**的。原因在于SQL标准中规定，NULL不等于任何值，包括它自身，所以在进行比较操作的时候，NULL相关的比较通常会被忽略掉。
>
> 此外，如果IN列表中所有的值都为空，则查询也不会返回任何行。这是因为IN查询的工作方式就是匹配给定的值，而空值并不在可匹配的范围内。
>
> 总的来说，在使用MySQL的IN查询时，要注意空值的处理，以避免因对空值的不当处理而导致查询结果不符合预期。

## mysql update 为什么不用跟select  insert delete 一样指定from

> MySQL的UPDATE语句设计之初就没有包含FROM子句，这是因为**MySQL不允许在UPDATE语句中使用子查询的结果来更新目标表**。
>
> 具体来说，MySQL中的UPDATE语句与其他SQL语句（如SELECT、INSERT、DELETE）有所不同，它不使用FROM子句。这是因为在早期的SQL标准中，当一个表被更新时，该表不能同时作为同一个查询中其他操作（比如子查询）的一部分。这样做是为了避免在一个未定义的状态中操作数据，防止在执行过程中出现不可预测的结果。例如，如果允许在同一个查询中既读取又修改同一张表，可能会引发数据的不一致性问题。
>
> 当需要根据另一张表的数据来更新当前表的数据时，可以使用多表更新（`JOIN`）的方式来实现。例如，通过`INNER JOIN`将另一张表与目标表连接起来，并基于连接条件来更新目标表的数据。
>
> 总的来说，由于设计上的限制和数据一致性的考虑，MySQL的UPDATE语句不支持直接使用FROM子句，但可以通过其他技术手段来实现类似的功能。

**在MySQL中，UPDATE语句并不需要使用FROM子句来指定要更新的表，因为UPDATE语句的语法已经提供了表名。MySQL的UPDATE语句的基本语法如下：**

```mysql
UPDATE table_name
SET column1 = value1, column2 = value2, ...
WHERE condition;
```

在这个语法中，`table_name` 指定了要更新的表名，`SET` 子句指定了要更新的列和它们的新值，`WHERE` 子句可选，用于指定更新的条件。因此，在MySQL中，通过直接指定表名来确定更新的表是很自然的。

与其他操作（如SELECT、INSERT、DELETE）不同，这些操作可能需要明确地指定要操作的表，因为它们可以从多个表中选择、插入或删除数据，因此需要明确指定操作的表。但是，UPDATE语句通常只涉及一个表，因此不需要使用FROM子句。

总之，在MySQL中，UPDATE语句的语法已经充分指定了要更新的表，因此不需要额外的FROM子句来指定表名。

## 	为什么join on 后面要查询条件不用加where

> 在 SQL 中，`JOIN` 用于将两个或多个表中的行连接起来，而 `ON` 关键字用于指定连接条件。`ON` 后面的条件用于确定哪些行应该连接起来，这通常是两个表之间共同的列或条件。而 `WHERE` 则是用于过滤结果集的条件。因此，在使用 `JOIN` 时，连接条件应该使用 `ON`，而其他用于筛选结果集的条件应该放在 `WHERE` 子句中。
>
> 举个例子，假设有两个表 `A` 和 `B`，我们想要根据它们之间的关联条件连接它们，并且在连接后的结果集中进行进一步的筛选，可以这样写：
>
> 这里，`ON` 子句指定了连接条件（`A.id = B.id`），而 `WHERE` 子句则用于进一步过滤结果集中的行（`A.some_column = 'some_value' AND B.another_column = 'another_value'`）。

```sql
SELECT *
FROM A
JOIN B ON A.id = B.id
WHERE A.some_column = 'some_value' AND B.another_column = 'another_value';
```

> ```
