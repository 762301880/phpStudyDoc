## 线上部署能不能用热更新[watcher](https://www.hyperf.wiki/2.2/#/zh-cn/watcher)

###  核心（重）

1. **本地开发环境**

   ✅ **强烈推荐使用 Watcher 热更新**

   修改代码自动重启，不用手动重启服务，开发效率极高

   

2. **线上生产环境**

   ❌ **绝对禁止使用 Watcher 热更新**

   生产必须**关闭热更新**，否则会导致：

   - 性能损耗
   - 代码异常、服务不稳定
   - 内存泄漏
   - 线上安全风险

### 开发环境怎么开启热更新（最简单）

```php
composer require hyperf/watcher --dev
```

启动命令改用

```php
php bin/hyperf.php server:watch
```

只要修改代码，**自动重启服务**，路由、控制器修改立刻生效。

### 线上生产环境 正确部署方式

1. 启动命令（生产标准）

```bash
php bin/hyperf.php start -d
```

`-d` = 守护进程后台运行

2. 生产更新代码流程

1. 上传新代码

2. 平滑重启不丢请求

   ```bash
kill -USR1 主进程ID
   # 或者
   kill -USR1 $(cat runtime/hyperf.pid)
   

   # 查看主进程id
cat runtime/hyperf.pid
   
   # 补充 
   ## 如果杀死了进程还是没生效 看一下是不是开了opcache
   php -i | grep opcache.enable_cli
   ```

✅ **生产唯一正确方案：代码更新 + 平滑重启**

不要用热更新！

### 为什么生产不能用热更新？

Hyperf 是**常驻内存框架**

热更新是**开发工具**，不是生产工具

文件监听会消耗服务器资源

生产环境代码不应该频繁变动，变动必须走平滑重启

## 方案二：零停机：多节点蓝绿更新（推荐，中小型生产）

适用：2 台以上机器 + Nginx 负载均衡，**用户无感知**，可回滚。

### 架构

```php
用户 → Nginx(upstream) → 节点A(旧)、节点B(旧)
```

### 步骤（逐台更新）

1. 从负载均衡摘除**节点 A**（Nginx 注释掉或设 weight=0）
2. 登录节点 A，执行 “方案一” 的 1–5 步（拉代码→重启）
3. 验证节点 A 健康（访问接口 / 健康检查）
4. 把节点 A 加回负载均衡
5. 重复 1–4 步更新节点 B

### Nginx 配置示例

```nginx
upstream hyperf_cluster {
    server 192.168.1.10:9501; # 节点A（更新时注释）
    server 192.168.1.11:9501; # 节点B
}

server {
    listen 80;
    location / {
        proxy_pass http://hyperf_cluster;
    }
}
```

✅ 优点：**零停机**、可回滚（切回旧节点）、简单可靠。
