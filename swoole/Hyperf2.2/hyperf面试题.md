## 请说一下hyperf生命周期

### Hyperf 生命周期核心解析

Hyperf 基于 Swoole/Swow 构建，采用**常驻内存 + 协程**模型，生命周期分为**应用启动**、**请求处理**、**进程 / 服务终止**三大阶段，与传统 PHP-FPM 框架差异显著。

## 一、核心差异：Hyperf vs 传统框架

| 阶段     | 传统 PHP-FPM 框架                | Hyperf 框架                        |
| -------- | -------------------------------- | ---------------------------------- |
| **启动** | 每次请求重新加载框架、配置、依赖 | 启动时**一次性**加载，常驻内存复用 |
| **请求** | 进程级隔离，请求结束销毁资源     | 协程级隔离，复用连接池、容器       |
| **资源** | 频繁创建 / 销毁连接              | 连接池复用，请求后归还             |

------

## 二、完整生命周期流程

### 1. 应用启动阶段（一次性）

**入口**：`php bin/hyperf.php start`

1. **环境初始化**：加载 `bootstrap.php`，定义常量、引入 Composer 自动加载、注册 AST 访问器（支持注解 / AOP）。
2. **容器构建**：初始化 DI 容器，加载配置（`config/autoload/`），扫描注解、路由、服务定义，构建容器依赖映射。
3. **服务器启动**：根据 `config/autoload/server.php` 配置启动 Swoole/Swow 服务器，创建 Master/Manager/Worker 进程，进入事件循环。
4. **资源预热**：初始化连接池（数据库、Redis、HTTP 客户端等），启动监听器、中间件、定时任务等核心组件。

### 2. 请求处理阶段（每次请求）

**以 HTTP 请求为例**

1. **请求接收**：Swoole 监听端口，接收请求，创建协程（C 栈开销极小），封装请求对象（`Psr\Http\Message\ServerRequestInterface`）。
2. **中间件流水线**：按配置顺序执行全局中间件、路由中间件，完成鉴权、日志、限流等预处理。
3. **路由解析**：匹配路由到对应控制器 / 方法，解析参数、绑定依赖注入。
4. **业务执行**：调用控制器方法，执行业务逻辑，通过协程安全调用 IO（数据库、Redis、RPC 等），底层自动挂起 / 恢复协程。
5. **响应返回**：生成响应对象，发送给客户端，**归还连接池资源**（如数据库连接），**清理协程上下文**（`Hyperf\Utils\Context`）。
6. **协程销毁**：请求结束，协程退出，内存不释放（进程常驻），等待下一次请求。

### 3. 终止阶段（服务停止 / 重启）

1. **进程终止**：接收停止信号（如 `kill`、`supervisor` 重启），触发 `WorkerStop`、`ServerStop` 事件，执行资源清理（关闭连接、持久化数据）。
2. **进程回收**：Master/Manager 进程回收子进程，释放系统资源，服务完全退出。

------

## 三、关键组件与生命周期事件

### 1. 核心组件生命周期

- **DI 容器**：默认**单例模式**（`get()` 获取），进程内全局共享，首次初始化后复用；`make()` 创建**短生命周期对象**，每次调用新建实例。
- **连接池**：启动时初始化，请求时获取连接，用完归还，进程内常驻，避免重复握手。
- **协程上下文**：`Hyperf\Utils\Context`，**请求级隔离**，存储用户会话、请求参数等，协程销毁时自动清理。

### 2. 关键生命周期事件（可监听插入自定义逻辑）

| 事件名            | 触发时机                   | 典型用途                           |
| ----------------- | -------------------------- | ---------------------------------- |
| `BootApplication` | 框架初始化完成，进程未创建 | 全局资源初始化、注册服务提供者     |
| `WorkerStart`     | Worker 进程启动            | 进程级初始化（如建立数据库长连接） |
| `OnRequest`       | 收到 HTTP 请求前           | 请求预处理（日志、鉴权、参数校验） |
| `WorkerStop`      | Worker 进程停止            | 资源清理、关闭连接池               |
| `ServerStart`     | 服务器启动完成             | 全局监控、初始化跨进程资源         |

------

## 四、开发避坑指南

1. **状态管理**：容器单例对象**禁止存储请求级状态**，状态应存于 `Context`；避免全局变量污染。
2. **资源清理**：在 `WorkerStop` 事件中主动关闭长连接、文件句柄，防止内存泄漏。
3. **协程安全**：IO 操作必须使用协程化组件（如 `hyperf/database`、`hyperf/redis`），避免阻塞事件循环。
4. **长时任务**：耗时任务需异步化（使用 `hyperf/async-queue`），避免阻塞 Worker 进程。
5. **进程隔离**：不同 Worker 进程内存独立，进程间通信需用 Redis、消息队列等方式。

------

## 五、总结

Hyperf 生命周期的核心是**常驻内存**与**协程复用**，启动阶段完成一次性初始化，请求阶段高效复用资源，终止阶段优雅清理。理解其生命周期是开发高性能、高并发协程应用的基础，需重点关注单例与协程上下文的隔离、资源的复用与清理。
