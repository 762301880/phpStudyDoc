## xss攻击

### XSS 攻击是什么？

**XSS = 跨站脚本攻击（Cross-Site Scripting）**

简单说：

**攻击者在你的网站里注入恶意 JavaScript 代码，让其他用户访问时自动执行。**

### 危害

- 偷用户 Cookie、登录态
- 跳转到钓鱼网站
- 窃取账号、篡改页面
- 劫持用户操作

### 两种常见类型

1. **反射型**：URL 里带恶意代码，用户点开就执行
2. **存储型**：恶意代码存进数据库，所有人访问都会触发（**最危险**）

### 模拟攻击

#### 恶意代码攻击

> 例如放到**laravel的public目录下  直接访问`url/xss.demo`**

```php
<?php
// 模拟用户提交的 XSS 攻击代码
$attack_code = '<script>alert("我是XSS攻击！偷Cookie啦！")</script>';
?>

<!DOCTYPE html>
<html>
<head>
    <meta charset="utf-8">
    <title>原生 PHP XSS 测试</title>
</head>
<body>
    <h2>❌ 1. 不防护（直接输出）→ 会被攻击！</h2>
    <?php
    // 危险！直接输出用户输入
    echo $attack_code;
    ?>

    <hr>

    <h2>✅ 2. 用 htmlspecialchars 防护 → 100% 安全！</h2>
    <?php
    // 安全！过滤转义后输出
    echo htmlspecialchars($attack_code, ENT_QUOTES);
    ?>
</body>
</html>
```



### Laravel 如何防护 XSS？

Laravel **默认自带强大的 XSS 防护**，几乎不用你手动写过滤！

下面是 Laravel 防护 XSS 的**3 个核心机制**：

1. Blade 模板默认自动转义（最重要）

Laravel 的 `{{ }}` 语法会**自动把 HTML/JS 转义成安全字符**。

安全写法（自动防 XSS）

```html
{{ $user->content }}  <!-- 自动转义 → 安全 -->
```

危险写法（不转义，会触发 XSS）

```html
{!! $user->content !!}  <!-- 原样输出 → 不安全 -->
```

**请求验证层**过滤不安全内容

```php
$request->validate([
    'name' => 'required|string|strip_tags', //strip_tags是自定义验证规则
]);
```

###  原生 PHP 防 XSS 3 种万能写法

#### 一、原生 PHP 防 XSS 最核心函数：

1. `htmlspecialchars()` （90% 场景都用它）

作用：把 **< > " ' &** 转成安全字符，让浏览器**不执行 JS 代码**。

攻击代码

```js
<script>alert('我是XSS攻击')</script>
```

不安全写法（会被攻击）

```php
echo $_GET['content']; // 直接输出 → JS 执行 → 弹窗弹出！
```

安全写法（防 XSS）

```php
echo htmlspecialchars($_GET['content']);
```

结果：

页面只会**显示文字**

```js
<script>alert('我是XSS攻击')</script>
```

**不会执行！不会弹窗！安全！**

#### 二、原生 PHP 防 XSS 3 种万能写法

1. 普通文本输出（最常用）

评论、用户名、输入框内容……

```php
echo htmlspecialchars($用户输入内容, ENT_QUOTES);
```

2. 输出到 HTML 属性里（如 input 的 value）

```php
$name = $_GET['name'];
echo '<input type="text" value="' . htmlspecialchars($name) . '">';
```

3. 富文本（必须过滤危险标签）

如果必须允许用户发图片、链接，用 `strip_tags()` 保留安全标签：

```php
$clean = strip_tags($用户内容, '<p><a><img><b><i>');
echo htmlspecialchars($clean);
```

####  三、最简单的原生 PHP 防 XSS 函数（直接复制用）

你可以把这个放到你的项目里，**所有输出都用它**：

```php
function xss($str) {
    return htmlspecialchars(trim($str), ENT_QUOT);
}
```

使用：

```php
echo xss($_POST['content']);
echo xss($row['title']);
```

#### 四、最直白总结（你一定能懂）

XSS 攻击原理：

黑客输入 JS 代码 → 你直接输出 → 代码执行 → 偷账号、偷 Cookie

原生 PHP 防护：

**所有用户输入输出到页面时，必须用 htmlspecialchars () 过滤！**

只要记住：

**不过滤 = 必被黑**

**过滤 = 100% 防住 XSS**

最终总结（原生 PHP 防 XSS 口诀）

1. **用户输入永远不要直接 echo**
2. **输出必须用 htmlspecialchars ()**
3. **富文本用 strip_tags () 过滤危险标签**

## csrf攻击

### 什么是csrf攻击

**CSRF** 全称 **Cross-Site Request Forgery，跨站请求伪造**，简单说：**利用你当前的登录状态，诱导你在不知情的情况下，在目标网站执行非自愿操作**。

### 一、核心原理

1. 你**正常登录**了网站 A（比如论坛、网银、后台），浏览器保留了登录 Cookie / 会话。
2. 你点开恶意链接 / 访问恶意页面 B。
3. 页面 B 偷偷发起**针对网站 A 的请求**，浏览器会**自动带上你在 A 的登录 Cookie**。
4. 网站 A 误以为是你本人操作，执行请求（改密码、转账、发帖子、删数据等）。

**关键前提**：用户处于**已登录状态** + 浏览器自动携带身份凭证（Cookie）。

### 二、通俗例子

1. 你登录了个人后台，没关网页。
2. 刷朋友圈点开一张 “搞笑图片”。
3. 这张图片实际是一段隐藏代码，自动请求接口：`https://后台.com/修改密码?new_pwd=123456`
4. 浏览器带着你的登录 Cookie 发送请求，后台直接把你密码改掉，你完全没察觉。

### 三、和 XSS 的区别（容易混淆）

- **XSS（跨站脚本）**：**偷你的身份 / 数据**，攻击者拿到你的账号权限。
- **CSRF**：**借用你的身份做事**，攻击者拿不到你的账号，只是冒用你的登录状态发请求。

### 四、常见防御方案（开发必用）

1. CSRF Token（最主流）

   每次表单 / 请求附带一个随机、一次性令牌，后端校验，恶意页面无法伪造。

2. 校验 Referer / Origin 请求头

   检查请求来源域名，拒绝非本站的请求。

3. Cookie 启用 SameSite 属性	

   `SameSite=Strict/Lax`，禁止跨站场景自动携带 Cookie（现代浏览器默认开启）。

4. 重要操作增加二次验证

   短信验证码、图形验证码、旧密码校验。

5. 敏感接口使用 POST + 不使用 GET 做增删改

   GET 请求最容易被图片、链接、iframe 触发，敏感操作禁用 GET。

### 五、普通用户如何防范

- 不在已登录重要网站时，乱点陌生链接、图片、弹窗。
- 用完网银、后台、账号后台**及时退出登录**。
- 开启浏览器安全策略，不随意允许跨站脚本。
