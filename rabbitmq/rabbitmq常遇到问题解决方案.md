## 如何保证消息不被重复消费（幂等性问题）

先说一个不太好听但很真实的结论：
 👉 **MQ天然是“至少一次（at-least-once）”投递，重复消费是常态，不是异常**

### 为什么会重复？

常见场景：

- 消费者处理完了，但**还没ack就挂了**
- 网络抖动，ack丢了
- RabbitMQ认为你没处理成功 → 重新投递

### 解决核心：**消费端必须幂等**

### 常见做法：

### 1️⃣ 业务唯一ID去重（最常用）

每条消息带一个唯一ID（如订单ID、消息ID）

消费时：

```
先查数据库/Redis：这个ID处理过没？
- 处理过 → 直接return（丢弃）
- 没处理 → 正常处理 + 记录ID
```

👉 Redis 示例：

> redis  [setnx](https://www.redis.net.cn/order/3552.html)

```php
SETNX msg:12345 1
```

### 2️⃣ 数据库唯一约束（硬核兜底）

比如：

```
UNIQUE(order_id)
```

重复插入直接报错 → 捕获异常 → 忽略

------

### 3️⃣ 状态机控制（适合复杂业务）

例如订单：

```
CREATED → PAID → SHIPPED
```

即使重复消费：

- 已经 PAID 的订单，再处理一次 → 不会重复扣款

##  如何防止消息丢失（可靠性问题）

丢消息通常发生在三段：

###  1. 生产者 → MQ

风险：

消息发了，但MQ没收到（网络/宕机）

✅ 解决：

**开启 Publisher Confirm（生产者确认机制）**

RabbitMQ：

```
channel.confirmSelect()
```

- 成功 → ack
- 失败 → nack → 重发

👉 这是必须开的，不然你连“发没发成功”都不知道

###  2. MQ 自身

风险：

MQ宕机 → 内存数据丢失

✅ 解决：

✔ 队列持久化

```
durable = true
```

✔ 消息持久化

```
deliveryMode = 2
```

👉 两个必须同时开！

###  3. MQ → 消费者

风险：

消费者还没处理完就挂了

 错误做法：

```
autoAck = true   ❌
```

你之前代码就是这个 👀 → 很危险

✅ 正确做法：

**手动ACK**

```
autoAck = false
```

处理流程：

```
1. 收到消息
2. 业务处理成功
3. 手动 ack
```

失败：

```
nack / requeue
```

## 如何保证消息顺序（顺序性问题）

这个问题最“反直觉”：
 👉 **MQ默认是“不保证全局顺序”的**

为什么顺序会乱？

- 多消费者并发消费
- 消费速度不同
- 重试机制打乱顺序

✅ 解决方案（看你要多严格）

1️⃣ 单队列 + 单消费者（最简单）

```
Queue → 1 Consumer
```

✔ 完全有序
 ❌ 吞吐量低

2️⃣ 按业务分队列（推荐）

例如：

```
order_1 → queue_1
order_2 → queue_2
```

👉 同一个订单保证顺序，不同订单并发

实现方式：

- routing_key 按 hash 分片

3️⃣ 消费端加锁 / 排序（复杂）

比如：

- Redis 分布式锁
- 本地队列排序

👉 成本高，一般不优先

4️⃣ RabbitMQ 特性（有限帮助）

RabbitMQ本身：

- 队列内是 FIFO
- 但**前提是单消费者**
